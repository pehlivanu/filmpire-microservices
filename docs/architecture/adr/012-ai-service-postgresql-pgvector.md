# ADR-012: ai-service Stores Conversations in PostgreSQL (pgvector), Not MongoDB

**Status:** Accepted
**Date:** 2026-07-23
**Deciders:** Project owner
**Amends:** ADR-002 (the AI row only — all other assignments stand)
**Related:** ADR-011 (self-healing read-through), ADR-004 ($0 budget)
**Issue:** #36

## Context

ADR-002 assigned MongoDB to ai-service, reasoning "conversation history and
embedding vectors; flexible, write-heavy." ARCHITECTURE.md §3.7 sketches the
domain accordingly: `Conversation(userId, type, messages, context, …)` and
`RecommendationModel(userId, …, embeddingVector, …)`.

ADR-011 then established how movie-service survives schema drift on MongoDB:
a document that no longer maps to the current model is discarded and re-fetched
from TMDB. That works **only because the movie catalog is re-derivable** — TMDB
is the seed source, so throwing a document away costs one HTTP call. ADR-011
says so explicitly, and warns the pattern must never be applied to user-owned
data.

Reviewing the remaining services against that boundary exposed a gap. Sorting
storage by whether drift is *caught* and whether data is *recoverable*:

| Service | Store | Drift caught at startup? | Data re-derivable? | Covered by |
|---------|-------|--------------------------|--------------------|------------|
| movie | MongoDB | No | **Yes** (TMDB) | ADR-011 self-healing |
| media | MongoDB | No | **Yes** (TMDB CDN refs) | ADR-011 self-healing |
| actor | PostgreSQL | **Yes** (Flyway + `validate`) | Yes | schema validation |
| user | PostgreSQL | **Yes** (Flyway + `validate`) | No | schema validation |
| **ai (planned)** | **MongoDB** | **No** | **No** | **nothing** |

ai-service as planned is the only cell with neither protection. Conversation
history is generated by users and exists nowhere else — it cannot be re-fetched
the way a movie can. So on MongoDB it would get schema-drift exposure with no
safe recovery: ADR-011's heal would silently **destroy real user conversations**,
and without Flyway nothing would catch the drift at deploy time either.

ai-service is currently a stub — one `HelloController`, no domain model, no
persisted data. The cost of changing this decision is zero today and rises
permanently once real conversation history exists.

## Decision

**ai-service uses PostgreSQL with the `pgvector` extension** for its own
database (`filmpire_ai`), under Flyway like the other relational services.

- Conversations and messages are relational tables with Flyway migrations and
  `ddl-auto: validate`, exactly as user-service manages accounts and favorites.
  User-owned data gets user-owned-data protections.
- Embedding vectors live in a `vector` column via pgvector, with an
  approximate-nearest-neighbour index for semantic search.
- Derived artifacts that are genuinely disposable (cached recommendation sets,
  scored candidate lists) may use Redis with a TTL — the existing cache tier —
  rather than a second persistent store.

This amends **only** the AI row of ADR-002. Movie and media stay on MongoDB;
their data is re-derivable, so ADR-011 covers them properly.

The resulting principle, which is the real output of this ADR:

> **User-owned data lives in PostgreSQL under Flyway. Re-derivable catalog data
> lives in MongoDB, where ADR-011's self-healing is safe.**

## Options Considered

**PostgreSQL + pgvector (chosen)** — conversations get ACID, constraints and
migration-checked schema like every other piece of user-owned data in the
system. pgvector handles embedding storage and similarity search comfortably at
this project's scale. Removes a storage engine from a service that has to run on
1–2 GB free-tier nodes (ADR-004), and PostgreSQL is already deployed and
operated. Cost: vectors in a relational column is less "web-scale" than a
dedicated vector store — irrelevant at this size.

**Keep MongoDB, add document schema-versioning** — read each document as a raw
`Document`, inspect a version field, migrate in code, then convert. This is the
honest way to run user data on a schemaless store, and it is what ADR-011's
`tmdbSyncVersion` *could not* be (that field is unreadable before
deserialization, which is why it was rejected there). Rejected: it means
hand-writing and testing a migration path for every model change, forever, on
the one service whose data can never be reconstructed if a migration is wrong.

**Split: conversations in PostgreSQL, embeddings in MongoDB** — technically
reasonable, and was the first instinct. Rejected as the worst of both: it keeps
the second engine and its operational cost while gaining nothing pgvector does
not already provide, and it splits one service's data across two stores for no
access-pattern reason.

**A dedicated vector database (Qdrant, Weaviate, Milvus)** — the "proper" answer
at scale. Rejected under ADR-004: another server to run locally and in K8s, on
nodes that are already tight, for a dataset measured in thousands of vectors.

## Consequences

- Easier: conversation history inherits the protections user data already has —
  Flyway migrations, `validate` at startup, ACID, foreign keys to nothing
  external (ADR-002 still forbids cross-service joins). One fewer engine on the
  free-tier node budget.
- Easier: the storage story becomes explainable in one sentence (see the
  principle above) instead of per-service trivia — which is worth something for
  a portfolio project whose point is demonstrating judgment.
- Harder: pgvector must be available in the Postgres image. The stock
  `postgres:17-alpine` does not ship it, so either the image becomes
  `pgvector/pgvector:pg17` or the extension is built in. This is a real change
  to `docker-compose.yml` and the K8s overlays, and it is the main cost of this
  decision.
- Harder: loses MongoDB's schemaless convenience for message payloads whose
  shape varies by provider. Mitigated with a `JSONB` column for provider-specific
  metadata — structured where structure exists, flexible where it does not.
- Note: this leaves MongoDB serving movie-service and media-service only. If
  media-service is later descoped or moved, the MongoDB dependency should be
  re-evaluated rather than kept out of habit.
- Revisit: if conversation volume ever outgrows a single PostgreSQL, or if
  vector search becomes the dominant workload rather than an accessory to it,
  a dedicated vector store becomes justifiable — this decision is scale-bound,
  not permanent.
